# v0.1 — Attributed live sessions

Plan for two parallel implementation tracks that together replace the
shared harbor password with per-host credentials and ship live
session recordings to FUNK storage with host attribution baked into
the archive filename.

The two tracks share a small contract (defined below) and are
otherwise independent.

---

## Contract — the surface both tracks code against

### Filename on disk (liquidsoap writes; daemon reads)

```
/var/funk/recordings/<mount>/<mount>-<YYYYMMDD>-<HHMMSS>.mp3
```

Liquidsoap does NOT try to embed attribution in the filename it
writes. Attribution is applied by the daemon at upload time, after
querying radio. This keeps `funk.liq` simple and avoids
filename-rename races inside the audio engine.

Examples:
- `/var/funk/recordings/live/live-20260526-184432.mp3`
- `/var/funk/recordings/breaking/breaking-20260526-191003.mp3`

### Filename in FUNK storage (daemon writes)

```
recordings/<mount>/<mount>-<YYYYMMDD>-<HHMMSS>-<safe_label>-<short_credential_id>.mp3
```

`safe_label` = lowercased, non-`[a-z0-9-]` collapsed to `-`, trimmed
to 32 chars. `short_credential_id` = first 8 chars of credential
UUID. If attribution is unknown (no matching session), the upload
key is `recordings/<mount>/<mount>-<YYYYMMDD>-<HHMMSS>-unattributed.mp3`.

Example: `recordings/live/live-20260526-184432-marisol-friday-show-c98f4805.mp3`

### Internal radio endpoints (radio owns; daemon + liquidsoap call)

**`POST /v1/radio/internal/harbor-auth`** — called by liquidsoap
on every harbor connection attempt.

Request body:
```json
{
  "mount": "live" | "breaking",
  "username": "<from broadcaster tool>",
  "password": "<from broadcaster tool>",
  "source_ip": "<peer IP, optional>",
  "connected_at": "2026-05-26T18:44:32Z"
}
```

Response (200):
```json
{ "valid": true, "credential_id": "c98f4805-...", "label": "Marisol Friday show" }
```

Response (200 with `valid: false`) means reject the connection.
Radio records the session as active until the matching disconnect.

**`POST /v1/radio/internal/harbor-disconnect`** — called by
liquidsoap when a harbor source drops.

```json
{ "mount": "live", "credential_id": "c98f4805-...", "disconnected_at": "2026-05-26T19:14:01Z" }
```

Radio marks the session inactive but retains the time-window record
for later attribution lookups (TTL: 7 days).

**`GET /v1/radio/internal/recording-attribution?mount=live&started_at=2026-05-26T18:44:32Z`**

Returns the session whose `[connected_at, disconnected_at]` window
contains `started_at` (with ±30s tolerance to account for liquidsoap
file-rotation timing).

Response (200):
```json
{ "credential_id": "c98f4805-...", "label": "Marisol Friday show" }
```

Response (404) if no matching session — caller treats as
"unattributed."

All three internal endpoints take a shared `RADIO_INTERNAL_SECRET`
bearer token (loopback / private-network use only; not exposed
publicly).

### Public API additions (radio owns)

**`POST /v1/radio/live/credentials`** — already exists in skeleton
form; this track makes it mint real per-host credentials instead of
echoing the shared password.

Returns:
```json
{
  "credential_id": "c98f4805-...",
  "label": "Marisol Friday show",
  "mount": "live",
  "harbor_host": "media.funk.example",
  "harbor_port": 8001,
  "username": "marisol-friday-show",
  "password": "<random 32-byte url-safe>",
  "expires_at": "2026-05-27T00:00:00Z"
}
```

**`DELETE /v1/radio/live/credentials/:id`** — already exists.
Revocation should be immediate: the next connection attempt with
that credential fails (200 with `valid: false`).

**`GET /v1/radio/recordings`** — already exists; daemon's work
makes `storage_url` populated. Response now includes
`credential_label` and `credential_id`.

`infra/services/liquidsoap/funk.liq`.

      username, password_hash, expires_at, revoked_at}>`. Acceptable

- Webhook to consumer on harbor connect/disconnect (consumer can
  poll `/v1/radio/live/status`).
- Per-credential rate limiting on harbor connection attempts.

## Track B — Recording upload daemon

Owned by Agent B. Lives in new directory `apps/recordings/`.

### Checklist

- [ ] New `apps/recordings/` service (Hono + Bun, mirror
      `apps/radio/` structure). Dockerfile, package.json, tsconfig.json,
      src/index.ts, src/env.ts.
- [ ] Service mounts `funk_recordings` volume read-write at
      `/var/funk/recordings`.
- [ ] FS watcher (chokidar or native `fs.watch`) over
      `/var/funk/recordings/{live,breaking}/`. On every file event,
      record `(path, mtime)`. Files become "stable" when their mtime
      hasn't moved for `STABILITY_SECONDS` (default 30).
- [ ] When a file becomes stable:
      1. Parse `mount` and `started_at` from the filename. Reject
         files that don't match the expected pattern.
      2. Query `GET http://radio:4003/v1/radio/internal/recording-attribution`
         with the shared `RADIO_INTERNAL_SECRET`. 404 → unattributed.
      3. Build the storage key per the contract above.
      4. Upload the file to FUNK storage via `POST /v1/files` with
         metadata `{credential_id?, label?, mount, started_at,
         duration_seconds}`. Use a daemon service credential
         (provisioned via `ADMIN_BOOTSTRAP_TOKEN`).
      5. On 2xx response with `storage_url`: delete the local file.
      6. On error: leave the file, log, schedule retry with
         exponential backoff (capped at 1 hour).
- [ ] `GET /health` endpoint returning `{status: "ok", pending: N,
      free_bytes: M}`. The `free_bytes` field is a basic disk-full
      early warning.
- [ ] Idempotent on restart: re-scan the volume at startup and
      re-process any stable files that haven't been deleted.

      Coordination note: this depends on the storage key being

### Out of scope for Track B

- Streaming uploads (full-file POST is fine; recordings are tens

- Compression / re-encoding (liquidsoap already writes mp3@128k).

## Integration — merge plan

Track A and Track B work in separate git worktrees. After both
complete:

1. Track A merges first (it owns the contract endpoints; Track B
   consumes them).
2. Conflicts will be limited to `infra/compose/compose.media.yml`
   (both add env vars to existing services and Track B adds a new
   service) and `tests/e2e/smoke.sh` (both extend it). Merge by
   keeping both changes.
3. Run `bash tests/e2e/smoke.sh` after merge. The combined smoke
   exercises both tracks end-to-end including the real harbor →
   recording → storage path.

## Why this sequencing

Track A defines the attribution semantics (sessions, credentials,
the lookup window). Track B consumes those semantics. They CAN
proceed in parallel because:

- The lookup endpoint contract is fully specified above. Track B
  treats it as an external API.
- During development, Track B can stub the attribution call (always
  return "unattributed") and still ship the daemon end-to-end.
  Integration happens at merge time.
- Track A doesn't need the daemon to test credentials — it
  validates harbor auth via a directly-driven ffmpeg client.

If the parallelization breaks down, the fallback is: ship Track A
alone (live credentials, no recording archive), then Track B in a
follow-up. Track B alone (without A) means recordings ship
unattributed forever — that's the worst order, hence the urgency
about doing them together or A-first.

## Non-goals for v0.1

- Live transcripts of harbor sessions (separate concern; lives in
  audio-service per ADR-0002).
- Per-credential bandwidth quotas.
- Multi-listener-stream tracking ("which credential is currently
  on-air right now" is already exposed via the existing
  `/v1/radio/now-playing` endpoint).
- Geofencing or IP allow-lists on harbor connections.
